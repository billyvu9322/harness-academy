## 1. Core Thesis

OpenAI's harness engineering thesis is:

> Humans steer. Agents execute.

The team intentionally built a real product with 0 manually-written code to force themselves to build the missing environment, tools, abstractions, feedback loops, and repo structure that Codex needed to do reliable work.

The main lesson is not "prompt harder." When Codex failed, OpenAI treated the failure as a harness defect:

- Missing capability.
- Missing tool.
- Missing abstraction.
- Missing guardrail.
- Missing documentation.
- Missing feedback loop.
- Missing mechanically enforced structure.

Their engineering role moved from writing code to designing the environment where agents can write, test, review, fix, and merge code with minimal human attention.

## 2. What OpenAI Built

### Product Built By Agents

OpenAI reports an internal beta software product built with 0 manually-written code.

Facts from the article:

- First commit landed in late August 2025.
- Article published February 11, 2026.
- Product has internal daily users and external alpha testers.
- Every line of code was written by Codex: application logic, tests, CI, docs, observability, internal tooling.
- Estimated build speed was about 1/10th the time of hand-written code.
- Repository grew to roughly one million lines across app logic, infrastructure, tooling, docs, and internal developer utilities.
- Roughly 1,500 PRs were opened and merged over five months.
- Initial team was three engineers driving Codex, later seven engineers.
- Throughput averaged about 3.5 PRs per engineer per day.
- Product had hundreds of internal users, including daily internal power users.

### Initial Repo Scaffold

The initial scaffold was generated by Codex CLI using GPT-5 and a small set of templates.

Generated parts included:

- Repository structure.
- CI configuration.
- Formatting rules.
- Package manager setup.
- Application framework.
- Initial `AGENTS.md`.

Important detail: even the agent instruction file was written by Codex.

### Agent-First Development Loop

OpenAI built a loop where engineers mostly interact through prompts.

Typical flow:

1. Human describes task.
2. Codex works in repo.
3. Codex opens a PR.
4. Codex reviews its own changes locally.
5. Codex requests specific agent reviews locally and in the cloud.
6. Codex responds to human or agent feedback.
7. Codex iterates until agent reviewers are satisfied.
8. Humans may review, but are not required for every PR.

OpenAI says review effort has moved increasingly toward agent-to-agent review.

### Standard Tool Access

Codex uses normal development tools directly.

Named tools and surfaces:

- `gh` for GitHub operations.
- Local scripts.
- Repository-embedded skills.
- Pull request feedback.
- Inline review comments.
- Push/update/squash/merge flows.

Humans do not copy-paste context into the CLI. The harness gives Codex direct routes to fetch context and act.

## 3. Application Legibility

OpenAI identified human QA capacity as a bottleneck. Their answer was to make the running application legible to Codex.

### Worktree-Isolated App Instances

They made the app bootable per git worktree.

Why this matters:

- One Codex task can run one isolated app instance.
- Multiple changes can be tested in parallel.
- Agent can validate the exact branch/worktree it is editing.
- Test servers, logs, and metrics can be torn down after task completion.

Official Codex docs confirm worktrees are used for independent tasks, background automations, handoff between local and worktree, and isolated Git state.

### Browser/DOM Harness

OpenAI wired Chrome DevTools Protocol into the agent runtime and created skills for:

- DOM snapshots.
- Screenshots.
- Navigation.
- UI interaction.
- Reproducing bugs.
- Validating fixes.
- Reasoning about rendered behavior.

The article's diagram says Codex can:

1. Select a target UI path.
2. Snapshot state before and after triggering UI behavior.
3. Observe runtime events through Chrome DevTools.
4. Apply fixes.
5. Restart.
6. Re-run validation until the app is clean.

Official Codex docs add related features:

- In-app browser for shared page view between human and Codex.
- Browser use for clicking, typing, inspecting rendered state, taking screenshots, downloading assets, and read-only page inspection JS.
- Browser comments for precise visual feedback.
- Chrome extension for signed-in websites and regular browser state.
- Appshots on macOS for sending the frontmost app window as visible context.

### Local Observability Stack

OpenAI exposed logs, metrics, and traces to Codex through a local ephemeral observability stack per worktree.

Article details:

- Logs, metrics, and traces are available to Codex.
- Stack is ephemeral for each worktree.
- Agents query logs with LogQL.
- Agents query metrics with PromQL.
- Figure alt text also names TraceQL for traces.
- App sends logs, metrics, and traces to Vector.
- Vector fans out to Victoria Logs, Victoria Metrics, and Victoria Traces.
- Codex queries, correlates, reasons, implements fixes, restarts, reruns workloads, tests UI journeys, and repeats.

Example goals become tractable:

- "Ensure service startup completes in under 800ms."
- "No span in these four critical user journeys exceeds two seconds."

Key idea: observability is not only for humans. It becomes agent-readable feedback.

### Long Runs

OpenAI reports single Codex runs working on one task for more than six hours, often while humans are sleeping.

This requires state, validation, and recovery to be inside the harness. A long run cannot depend on continuous human babysitting.

## 4. Repository Knowledge As System Of Record

OpenAI rejected the "one giant AGENTS.md" pattern.

Why it failed:

- Context is scarce, huge instructions crowd out task/code/relevant docs.
- Too much guidance becomes non-guidance.
- Giant docs rot quickly.
- A monolithic blob is hard to verify mechanically.

Replacement pattern:

- Short `AGENTS.md`, around 100 lines.
- `AGENTS.md` acts as table of contents, not encyclopedia.
- Structured `docs/` directory becomes system of record.
- Deeper docs hold product, design, architecture, plans, quality, and decision context.

OpenAI's repo knowledge structure includes:

- Design documentation catalogued and indexed.
- Verification status for designs.
- Core beliefs defining agent-first operating principles.
- Architecture documentation with top-level map of domains and package layering.
- Quality document grading product domains and architectural layers.
- Active plans.
- Completed plans.
- Known technical debt.
- Progress logs.
- Decision logs.

OpenAI enforces docs mechanically:

- Dedicated linters validate the knowledge base.
- CI jobs check that docs are up to date, cross-linked, and structured correctly.
- A recurring doc-gardening agent scans for stale or obsolete docs and opens fix-up PRs.

## 5. Progressive Disclosure

OpenAI's context strategy is progressive disclosure.

The agent starts with a small stable entrypoint, then follows links to deeper context only when needed.

Principle:

> Give Codex a map, not a 1,000-page instruction manual.

This maps directly to harness construction:

- Root file: entrypoint.
- Structured docs: deeper truth.
- Plans: task-specific operating state.
- Linters: keep docs trustworthy.
- Agents: read only relevant context by phase.

Official Codex `AGENTS.md` docs confirm this layering pattern:

- Codex discovers global guidance from `~/.codex/AGENTS.md` or `AGENTS.override.md`.
- Codex discovers project guidance from root down to current working directory.
- Closer files override broader guidance because they appear later.
- Codex has a default combined size cap, 32 KiB by default.
- Nested `AGENTS.md` or `AGENTS.override.md` files support subproject-specific rules.

The public AGENTS.md standard reinforces the same pattern:

- README is for humans.
- AGENTS.md is a predictable place for agent-specific setup, tests, conventions, security notes, PR instructions, and subproject guidance.
- Large monorepos can use nested AGENTS.md files.
- At time of writing, the AGENTS.md site says OpenAI's main repo has 88 AGENTS.md files.

## 6. Agent Legibility

OpenAI optimizes the repo for agent legibility.

From Codex's point of view:

- If it is not accessible in context, it effectively does not exist.
- Google Docs, Slack threads, and tacit human memory are invisible unless encoded into repo artifacts.
- Repo-local, versioned artifacts are what agents can inspect and act on.

Legible artifact types:

- Code.
- Markdown.
- Schemas.
- Executable plans.
- Architecture docs.
- Decision logs.
- Quality grades.
- Tests.
- Custom lints.

Technology selection also follows agent legibility:

- Prefer dependencies and abstractions that can be fully internalized and reasoned about in repo.
- "Boring" technologies are often easier for agents due to composability, API stability, and training-set representation.
- Sometimes it is cheaper to reimplement a small subset than work around opaque upstream behavior.

Article example:

- Instead of a generic `p-limit` style package, the team implemented a custom `map-with-concurrency` helper.
- It is integrated with OpenTelemetry instrumentation.
- It has 100% test coverage.
- It behaves exactly as their runtime expects.

## 7. Architecture And Taste Enforcement

OpenAI says documentation alone is insufficient for a fully agent-generated codebase. They enforce invariants mechanically.

### Layered Domain Architecture

OpenAI uses a rigid architectural model.

Within each business domain, code can only depend forward through layers:

```txt
Types -> Config -> Repo -> Service -> Runtime -> UI
```

Cross-cutting concerns enter through one explicit interface:

```txt
Providers
```

Cross-cutting examples:

- Auth.
- Connectors.
- Telemetry.
- Feature flags.

Anything outside allowed dependency edges is disallowed and mechanically enforced.

### Custom Linters And Structural Tests

OpenAI enforces architecture with:

- Custom linters.
- Structural tests.
- Static checks for structured logging.
- Naming conventions for schemas and types.
- File size limits.
- Platform-specific reliability requirements.
- Taste invariants.

Important harness tactic:

- Lint error messages are written to inject remediation instructions into agent context.

This turns lint failures into self-healing feedback. The agent sees not only "wrong," but what to do next.

### Constraints As Multipliers

OpenAI's rule:

- Enforce boundaries centrally.
- Allow autonomy locally.

They are not trying to micromanage each implementation. They enforce correctness, boundaries, reproducibility, and legibility. Inside those boundaries, agents can choose implementation details.

OpenAI explicitly accepts that generated code may not match human stylistic preference if it is correct, maintainable, and legible to future agents.

## 8. Merge Philosophy Under High Throughput

OpenAI says conventional merge norms change when agent throughput exceeds human attention.

Their repo operates with:

- Minimal blocking merge gates.
- Short-lived pull requests.
- Follow-up runs to address flaky tests instead of blocking indefinitely.
- Cheap correction loops.

OpenAI warns this would be irresponsible in low-throughput environments. In their high-throughput harness, waiting can cost more than correction.

This philosophy depends on the rest of the harness:

- Fast PRs.
- Mechanical checks.
- Follow-up agents.
- Continuous cleanup.
- Low-cost remediation.

## 9. What Agent-Generated Means At OpenAI

OpenAI says "agent-generated" means everything in the codebase.

Agents produce:

- Product code and tests.
- CI configuration and release tooling.
- Internal developer tools.
- Documentation and design history.
- Evaluation harnesses.
- Review comments and responses.
- Scripts that manage the repository itself.
- Production dashboard definition files.

Humans remain in the loop, but at a higher abstraction level:

- Prioritize work.
- Translate user feedback into acceptance criteria.
- Validate outcomes.
- Identify missing tools, guardrails, and docs when agents struggle.
- Feed fixes back into the repo, with Codex writing those fixes.

## 10. Increasing Autonomy

OpenAI says the repo recently crossed a threshold where Codex can drive a feature end-to-end from a single prompt.

Given one prompt, Codex can:

- Validate current codebase state.
- Reproduce a reported bug.
- Record a video demonstrating the failure.
- Implement a fix.
- Validate the fix by driving the app.
- Record a second video demonstrating resolution.
- Open a PR.
- Respond to agent and human feedback.
- Detect and remediate build failures.
- Escalate to humans only when judgment is required.
- Merge the change.

OpenAI cautions that this depends heavily on the repo's specific structure and tooling. It should not be assumed to generalize without similar investment.

## 11. Entropy Management And Garbage Collection

OpenAI found that full agent autonomy creates new entropy problems.

Cause:

- Codex replicates existing repo patterns.
- If patterns are uneven or suboptimal, agents copy them.
- Drift compounds over time.

Initial human cleanup did not scale:

- Team used to spend every Friday, 20% of the week, cleaning up "AI slop."

Replacement pattern:

- Encode "golden principles" directly into repo.
- Run recurring cleanup processes.
- Background Codex tasks scan for deviations.
- Background tasks update quality grades.
- Background tasks open targeted refactoring PRs.
- Most cleanup PRs can be reviewed in under a minute and automerged.

Golden principle examples:

- Prefer shared utility packages over hand-rolled helpers so invariants stay centralized.
- Do not probe data YOLO-style.
- Validate boundaries or rely on typed SDKs so agents do not build on guessed shapes.

OpenAI describes this as garbage collection:

- Technical debt is paid down continuously in small increments.
- Human taste is captured once, then enforced continuously.
- Bad patterns are caught daily before spreading.

## 12. Execution Plans

OpenAI's Cookbook describes `PLANS.md` and ExecPlans for multi-hour Codex problem solving.

Purpose:

- Let Codex implement complex tasks that require research, design, and implementation.
- Give users a plan they can review before a long implementation.
- Support multi-hour autonomous work from a single prompt.

OpenAI says a similar plan has enabled Codex to work for more than seven hours from one prompt.

Key ExecPlan requirements:

- Fully self-contained.
- Living document.
- Must enable a complete novice to implement end-to-end from current working tree and the plan alone.
- Must produce demonstrably working behavior.
- Must define terms of art in plain language.
- Must include purpose, progress, surprises/discoveries, decision log, outcomes/retrospective, context, plan of work, concrete steps, validation, recovery, artifacts, interfaces, and dependencies.
- Must be updated as progress, discoveries, and decisions happen.

Harness interpretation:

- ExecPlans are a state subsystem.
- They solve long-horizon continuity.
- They reduce hidden context.
- They make task execution inspectable by humans and future agents.

## 13. Skills

Official Codex docs define skills as task-specific capabilities packaged as instructions, resources, and optional scripts.

OpenAI's Codex skill model:

- Skills are reusable workflows.
- Skills are available in CLI, IDE extension, and Codex app.
- Skills use progressive disclosure.
- Codex starts with skill name, description, and file path.
- Full `SKILL.md` is loaded only when the skill applies.
- Initial skill list is capped at roughly 2% of model context window, or 8,000 chars when unknown.

Skill directory shape:

```txt
my-skill/
  SKILL.md      # required
  scripts/      # optional
  references/   # optional
  assets/       # optional
  agents/       # optional metadata/dependencies
```

Harness interpretation:

- Skills turn repeated human guidance into reusable agent workflows.
- They keep root prompts small.
- They reduce repeated failures by giving agents exact process rules.
- They can declare tool dependencies such as MCP servers.

## 14. Subagents

Official Codex docs describe subagents for parallel specialized workflows.

Codex supports:

- Spawning specialized agents in parallel.
- Routing follow-up instructions.
- Waiting for results.
- Closing agent threads.
- Custom agents with separate model config, instructions, sandbox mode, MCP servers, and skills.

Built-in agent types in docs:

- `default`: general-purpose fallback.
- `worker`: execution-focused implementation/fix agent.
- `explorer`: read-heavy codebase exploration agent.

Useful limits and controls:

- Subagents only spawn when explicitly asked.
- They inherit current sandbox policy.
- `agents.max_threads` defaults to 6.
- `agents.max_depth` defaults to 1 to prevent recursive fanout.
- Custom agents can be read-only for review/exploration or workspace-write for implementation/debugging.

Harness interpretation:

- Subagents are useful for review, exploration, UI debugging, and parallel audits.
- They need sandbox and depth controls to avoid runaway cost and unpredictable delegation.

## 15. Sandboxing And Approvals

Official Codex docs define sandbox as the boundary that lets Codex act autonomously without unrestricted machine access.

Sandbox controls:

- Files Codex can read or write.
- Whether commands can use network.
- Whether spawned commands inherit boundaries.
- When Codex must ask for approval.

Key distinction:

- Sandbox defines technical boundaries.
- Approval policy decides when Codex pauses before crossing them.

Common sandbox modes:

- `read-only`: inspect only, no edits or commands without approval.
- `workspace-write`: read, edit within workspace, run routine commands inside boundary. This is the default low-friction local mode.
- `danger-full-access`: no sandbox restrictions, only for intentional full access.

Approval policies:

- `untrusted`: asks before commands outside trusted set.
- `on-request`: works in sandbox, asks when it needs more.
- `never`: does not stop for approval prompts.

OpenAI docs recommend lower-risk automation with:

```txt
sandbox_mode = workspace-write
approval_policy = on-request
```

Full access means:

```txt
sandbox_mode = danger-full-access
approval_policy = never
```

Harness interpretation:

- Autonomy without sandbox is not harness engineering.
- The harness must separate low-risk routine work from risky operations.
- Approval fatigue is reduced by letting safe work proceed inside explicit boundaries.

## 16. Automations

Official Codex docs describe automations for recurring background tasks.

Capabilities:

- Run recurring tasks in background.
- Add findings to inbox/triage.
- Archive task if nothing to report.
- Run project-scoped automations in local project or new worktree.
- Combine automations with skills.
- Create/update automations from a normal Codex thread.
- Thread automations can preserve thread context.
- Standalone/project automations can run independent scheduled checks.

Security model:

- Automations use default sandbox settings.
- Read-only mode blocks file/network/app actions needing writes.
- Workspace-write blocks outside-workspace writes and network unless rules allow.
- Full access background automation is elevated risk.
- Managed organizations can restrict sandbox modes and prevent `approval_policy = never`.

Harness interpretation:

- Automations are the recurring operations layer.
- They turn repeated checks, PR babysitting, stale-doc cleanup, bugfix search, and project briefings into scheduled agent work.

## 17. OpenAI's Harness Stack As A System

OpenAI's system can be summarized as these layers:

```txt
Human intent
  -> Prompt / acceptance criteria
  -> AGENTS.md table of contents
  -> Structured repo knowledge base
  -> ExecPlan for complex work
  -> Codex agent / subagents / skills
  -> Worktree-isolated execution environment
  -> Browser, DOM snapshots, screenshots, app driving
  -> Logs, metrics, traces via local observability stack
  -> Custom linters, structural tests, CI, type checks
  -> Agent reviews and feedback loops
  -> PR, merge, cleanup, doc gardening
  -> Updated repo knowledge and golden principles
```

## 18. Practical Takeaways For Our Harness Template

### Must-Have Artifacts

- Short `AGENTS.md`, `CLAUDE.md`, or `GEMINI.md` as table of contents.
- Structured `docs/` knowledge base.
- Architecture map.
- Quality/grade tracker.
- Test matrix.
- Execution plans for large work.
- Progress and decision logs.
- Feature tracking in root context file for seminar model.
- Trace/evidence records.
- Recurring cleanup backlog.

### Must-Have Tools

- File and code search.
- Git/PR tooling.
- Local test/build commands.
- Browser/DOM/screenshot tooling.
- Worktree isolation.
- Logs/metrics/traces query access.
- Custom lints with remediation messages.
- Agent review workflows.
- Scheduled background automations.

### Must-Have Policies

- No completion without evidence.
- Root context file is map, not encyclopedia.
- Any invisible context must be encoded into repo artifacts.
- Architecture boundaries are enforced by code, not only prose.
- Taste is captured once and enforced repeatedly.
- Merge gates can be lighter only if correction loops are strong.
- Risky operations require sandbox/approval controls.
- Repeated agent friction becomes harness backlog.

## 19. What Is Still Unknown

OpenAI explicitly says they are still learning:

- How architectural coherence evolves over years in fully agent-generated systems.
- Where human judgment adds the most leverage.
- How to encode human judgment so it compounds.
- How the system evolves as models become more capable.

Their current conclusion:

- Software still demands discipline.
- The discipline moves from hand-writing code into scaffolding, tooling, abstractions, feedback loops, and control systems.

## 20. One-Sentence Summary

OpenAI applies harness engineering by turning the repository into an agent-readable, mechanically enforced operating system where Codex can plan, code, run the app, observe UI/logs/metrics/traces, review, fix, merge, clean up entropy, and improve the harness itself while humans steer intent and validate outcomes.
